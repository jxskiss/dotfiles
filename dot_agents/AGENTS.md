# Commenting Rules

- Do not add redundant comments that merely restate what the code does.
- Do not add noisy comments for obvious assignments, branches, loops, imports, or function calls.
- Do not use comments as filler, narration, or decoration.
- Do not leave stale comments, commented-out code, TODOs without owner/context, or comments that contradict the implementation.
- Prefer clear names and simple structure over comments when the code can explain itself.
- Write comments only when they explain why the code exists, why a non-obvious approach is necessary, or what constraint must be preserved.
- Good comments should capture intent, tradeoffs, invariants, edge cases, external requirements, performance/security concerns, or historical context that is not obvious from the code.
- Keep comments concise, specific, and close to the code they clarify.
- Update or remove related comments whenever changing behavior.
